# CLAUDE.md — conventions for agents working in this repo

`shakespeare-nanogpt` is a character/BPE-level GPT (vendored [nanoGPT](https://github.com/karpathy/nanoGPT))
trained on Shakespeare, plus an ongoing LLM-assisted research series (Claude as
an automated researcher under human direction) and the docs/charts that explain
it. Start with [`README.md`](README.md) for the project itself and
[`MODELS.md`](MODELS.md) for the versioned model series.

This file is the standing guidance for anyone (human or agent) editing the repo.
Two conventions matter enough to call out up front: **how charts are made** and
**what the design system is called**.

## Charts: always use the `dataviz` pipeline

**Every chart in this repo — without exception — is generated by the
[`dataviz/`](../../tools/dataviz/README.md) pipeline.** That includes charts in the
[README](README.md), the [model cards](../../research-docs/model-cards/), the
[experiment reports](../../research-docs/reports/), and the [leaderboard](./leaderboard.md),
plus any new chart in any future doc.

Rules:

- **Never hand-author chart SVGs**, and never reach for matplotlib, Chart.js,
  D3, screenshots of spreadsheets, or any other charting tool. If a doc needs a
  chart, it comes from `dataviz`.
- **To add or change a chart**, edit [`dataviz/build.py`](../../tools/dataviz/build.py)
  (data + chart definitions live there), then run `uv run python tools/dataviz/build.py`
  from the repo root or `cd tools/dataviz && python3 build.py`. It writes self-contained
  HTML to `../../tools/dataviz/output/` and exports light/dark PNGs into
  [`research-docs/reports/assets/`](../../research-docs/reports/assets/).
- **Embed both themes.** In Markdown that supports it (the reports), use a
  `<picture>` with light/dark `srcset` so charts follow the reader's color
  scheme. Elsewhere (README, model cards) embed the `.light.png`.
- The renderer is pure Python 3 stdlib (`../../tools/dataviz/dataviz.py`); only the PNG
  export step shells out to headless Chrome (skippable with `--no-png`).

When in doubt: if you're about to put a number in a chart, it goes through
`dataviz`.

## Design system: "Vercel Geist"

The `dataviz` visual language is our own chart design system, **derived from
Vercel Geist** — Vercel's design system:

- light: <https://vercel.com/design.md>
- dark: <https://vercel.com/design.dark.md>

Naming convention for docs and code comments: call it **"Vercel Geist"** (or
"Vercel's Geist design system"), **not** bare "Geist" and **not** "Geist by
us." Always attribute it to Vercel and, on first mention in a doc, link both the
light (`design.md`) and dark (`design.dark.md`) sources above. It is explicitly
**not** Anthropic's dataviz system — we borrowed the *engineering* approach
(self-contained HTML + inline SVG, nice-axis ticks, a vertical layout stack),
but every visual token (Geist Sans, color ramps, 6px radius, weight-600
headings) is Vercel Geist. Tokens live in
[`dataviz/designsystem/tokens.py`](../../tools/dataviz/designsystem/tokens.py).

## Other repo conventions

- **Trained weights are never committed** (gitignored). Everything needed to
  *reproduce* them is — see the rebuild commands in [`MODELS.md`](MODELS.md).
- **Reports are frozen.** [`research-docs/reports/experiment-NN.md`](../../research-docs/reports/README.md) files
  are never edited after publishing; supersede with a new number. The *living*
  records are [`leaderboard.md`](./leaderboard.md),
  [`MODELS.md`](MODELS.md), and the [model cards](../../research-docs/model-cards/).
- **The LLM-assisted research experiment is split across the monorepo.** The
  shared engine lives in [`../../core/`](../../core/): the round-running scripts
  (`../../core/eval/eval.py`, `../../tools/claude_cost.py`) and the nanoGPT
  training/sampling code (`../../core/nanogpt_core/`). This project holds the
  experiment's data and outputs: the per-round run configs/outputs under
  [`runs/`](runs/) and [`config/`](config/), the datasets under [`data/`](data/),
  the held-out test set ([`test.txt`](test.txt)), and the
  [leaderboard](./leaderboard.md). New shared engine code goes in `../../core/`;
  new datasets, configs, and run outputs go here.
- **Models are a living series** (`shakespeare-nanogpt-N`), each pinned to a git
  tag and scored on the same fixed held-out test (`test.txt`) in
  bits-per-character (BPC).
- **Each released version has a frozen, self-contained folder under
  [`models/`](models/README.md)** (`models/shakespeare-nanogpt-N/`): a runnable
  snapshot — `model.py`, `config.py`, `train.py`, `sample.py`, `eval.py`,
  `prepare.py`, `configurator.py` — copied from whatever produced it, so the
  version stays reproducible even as the shared engine in `../../core/` changes.
  Duplication between the two trees is intentional; when a research round becomes
  a release, snapshot its code into a new `models/` folder rather than pointing at
  the evolving core engine code. Generated artifacts in these folders (`*.pt`,
  `*.bin`, `*.pkl`, `input.txt`, `raw.txt`) are gitignored.
