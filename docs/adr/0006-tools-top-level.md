# ADR 0006: `tools/` is a top-level home for researcher tooling

- **Status:** Accepted
- **Date:** 2026-06-27
- **Deciders:** Romello Goodman (with Claude)

## Context

Two pieces of tooling needed a home: the dataviz chart pipeline (every chart in the
repo is generated by it) and `claude_cost.py` (session-cost accounting for the
research rounds). Neither is shipped engine code, and neither is a Claude *skill*.

The original monorepo plan folded dataviz under `skills/`. But these are not agent
capabilities the model invokes — they are tools the agent *operates as the
researcher*: it runs them to produce charts and to account for what a round cost. Put
them under `skills/` and you blur the line between "things Claude can do" and "things
the researcher runs," and you lose a clear answer to "where does a chart come from?"

## Decision

We will give researcher tooling its own top-level peer, `tools/`. It holds the
tooling the agent operates while doing research — `tools/dataviz/` (the chart
pipeline) and `tools/claude_cost.py` (cost accounting) — kept distinct from Claude
*skills* (agent capabilities) and from shipped engine code in `core/`.

Every chart in the repo goes through `tools/dataviz/` — README charts, model-card
charts, experiment-report charts, the leaderboard. There is no second way to make a
chart.

## Consequences

- A clean three-way separation that maps to three different jobs:
  - `core/` — the model engine (shipped, imported).
  - `tools/` — researcher tooling (operated, not shipped).
  - skills — agent capabilities.
- Charts have exactly one canonical source, so a reader always knows where a chart
  came from and a writer always knows where to add one.

## Alternatives considered

- **Put it under `skills/`** (the original plan) — rejected. These are tools the
  researcher operates, not capabilities the agent exposes; the categories are
  different and conflating them costs the clean separation above.
- **Put it under `core/`** — rejected. It is not the model engine; a project doesn't
  `import` the chart pipeline. Folding it into `core/` would muddy what `core/` is.
